# Proposal: Top-5 next PRs for the moku plugin (2026-05)

> **Status: implemented in v0.28.0.** PR1 (adversarial verify + `moku-build-wave`/`moku-migrate-sweep`
> workflows + `moku-skeptic`), PR2 (STATE.md Recovery block + `memory-schema.md` + next/status fast
> path), PR3 (documented the path-based tool posture in `tool-scoping.md` — `disallowed-tools` does
> not fit moku's path gates), PR4 (`hook-patterns.md` + self-correcting guard reasons), and PR5
> discovery polish (`SKILL-INVENTORY.md`, `/moku:check usage`, `$schema`/`displayName`,
> `effort` tiers; monolith split evaluated and **deferred** — see `docs/plugin-composition-evaluation.md`).

Derived from current Claude Code / Anthropic capabilities (research preview of dynamic
workflows + Opus 4.8, the 1M-token window now at standard pricing, Skills, the memory tool,
hooks `continueOnBlock`, and plugin discovery/dependency tooling). Each item is scoped to a
single PR. Ordered by leverage-to-effort.

This release (v0.27.0) already landed the foundation these build on: the vendored spec +
fast index injected everywhere, the `.planning/` no-commit rule, the cleanup command, the
1M-context adjustments, and two shipped dynamic workflows (`/moku-verify`, `/moku-audit`).

## PR1 — Dynamic workflows for the remaining fan-out phases
**Why now.** v0.27.0 ships `moku-verify` and `moku-audit`. The same orchestration model fits
the other embarrassingly-parallel phases.
**Scope.**
- `workflows/moku-build-wave.js` — drive a single build wave: builders in parallel
  (worktree isolation per plugin to avoid write conflicts) → verify-as-each-completes
  (pipeline, not barrier) → wave-judge. Mirrors `build-wave-execution.md`.
- `workflows/moku-migrate-sweep.js` — discover migration sites → transform each in an isolated
  worktree → verify. For the `migrate` verb's large file sweeps.
- Adopt the **adversarial-verification** pattern in `moku-verify`: spawn N skeptics per
  blocker that try to *refute* it; drop findings that don't survive. Cuts false positives.
**Risk.** Research-preview API; no mid-run gating — keep these for non-interactive phases only.
**Effort.** Medium. Reuses existing agents as `agentType`.

## PR2 — Memory tool + multi-session resumability for STATE.md
**Why now.** The API memory tool + context-awareness enable robust cross-session recovery; our
STATE.md already approximates this by hand.
**Scope.**
- Define a `.planning/memory/` convention the validators' `memory: user` already gesture at,
  and document the multi-session recovery pattern (fast reconstruct from STATE.md + learnings).
- Add a "recovery header" to STATE.md (last-good wave, open blockers, next action) optimized for
  a cold session to rehydrate in one read.
- Teach `/moku:next` and `/moku:status` to reconstruct from this header first.
**Risk.** Low — additive to existing STATE.md schema.
**Effort.** Low–Medium.

## PR3 — Per-stage tool scoping via Skills `disallowed-tools`
**Why now.** Skills can now remove tools from context while active (v2.1.152), a cleaner control
than hook-based blocking for "what's even allowed in this stage."
**Scope.**
- During brainstorm, disallow `Write`/`Edit` outside `.planning/` natively (today this is a hook,
  `brainstorm-guard.sh`) — keep the hook as a backstop but make the intent declarative.
- Scope validator agents to read-only toolsets via the skill, not just frontmatter.
- Audit each command's `allowed-tools` against what the stage actually needs.
**Risk.** Low. Hooks remain as defense-in-depth.
**Effort.** Low.

## PR4 — Informative hooks via `continueOnBlock`
**Why now.** PostToolUse `continueOnBlock` (v2.1.139+) feeds a rejection reason back to Claude so
it self-corrects, instead of a hard stop the model can't see.
**Scope.**
- Convert `approve-planning-writes.sh`, `brainstorm-guard.sh`, and the new `.planning/` commit
  guard in `verify-before-commit.sh` to return structured reasons via `continueOnBlock` so Claude
  retries correctly (e.g. "write to `.planning/` instead of `src/`").
- Move hook scripts to the `args: string[]` exec form to remove shell-escaping fragility.
**Risk.** Low. Behavior-preserving; better UX.
**Effort.** Low.

## PR5 — Plugin composition + discovery polish
**Why now.** Plugin dependency enforcement, richer `/plugin` discovery, and `claude plugin details`
(per-session token cost) reward well-factored, well-described plugins.
**Scope.**
- Evaluate splitting the monolith into composable units (e.g. `moku-core`, `moku-audit`,
  `moku-web`) with declared dependencies, so consumers install only what they need.
- Curate component inventories + descriptions so the Discover/Browse screens read well.
- Add a `/moku:check --usage` view that surfaces the plugin's measured token footprint via the
  `/usage` per-category breakdown (v2.1.149).
**Risk.** Medium — splitting touches the manifest and marketplace entries; do it behind a minor
version and keep a meta-package for the full bundle.
**Effort.** Medium–High.
