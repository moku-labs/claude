# Moku Common Conventions (`@moku-labs/common`) ‚Äî authoritative rules

These are the **family-level conventions** for consuming the shared `@moku-labs/common` package.
They are repo-owned and citable: validators (`moku-common-validator`), the `validate-common-usage`
hook, and human reviewers reference them by the stable IDs below (**MC1‚ÄìMC3**). They are *separate*
from the upstream Moku Core invariants R1‚ÄìR8 (which live in `spec/11-INVARIANTS.md` and govern the
kernel/plugin contract, not how a project consumes the common package).

> This file is **repo-owned** and lives under the `moku-common` skill on purpose: the vendored
> `spec/**` tree is re-generated by `spec-sync` and would clobber anything added there. A one-line
> pointer in `skills/moku-core/references/house-style.md` marks these rules as approved/required so
> other validators treat them as house style, not a per-project invention.

## Severity & scope conventions

- Use the standard severities from `agent-preamble.md`: **BLOCKER** (must fix), **WARNING** (should
  fix), **INFO** (suggestion). Cite the rule ID (`MC1`/`MC2`/`MC3`) in every finding.
- These rules apply to **project source that consumes the package** ‚Äî framework plugins (Layer 2),
  consumer plugins (Layer 3), CLI entry points, and `scripts/`. They do **not** apply to the
  `@moku-labs/common` package's own source (the brand kit, the env provider, the log sink are the
  *implementations* of these surfaces).
- **Prefer not firing over a false positive.** When uncertain whether a usage is one of the
  documented exceptions, downgrade to WARNING (validator) or do not block (hook).

## Shared exceptions (apply to all three rules)

Never flag these ‚Äî they are allowed by construction:

1. **Test files** ‚Äî `*.test.ts`, `*.spec.ts`, `*.test.tsx`, `*.spec.tsx`, anything under
   `**/__tests__/**`, and test infrastructure (`*.mock.ts`, `*.fixture.ts`, `vitest.setup.ts`,
   `*.config.ts`). Tests legitimately use `console.*`, `process.env`, and raw output.
2. **The brand-kit source** ‚Äî the `@moku-labs/common/cli` implementation, path-matched as
   `*/common/src/cli/*`. It *is* the ANSI/box/spinner implementation (MC1 does not apply to it).
3. **The documented log sink** ‚Äî the single lowest-level transport that `ctx.log` ultimately writes
   through MAY call `console.*`. Mark it with a `// @log-sink` comment on or just above the call so
   tooling recognizes it (MC2 does not apply to a marked sink).
4. **Env providers** ‚Äî the module backing `envPlugin` (path-matched as `*/env/*`, or a file
   exporting a `*EnvProvider` / named `env-provider.ts`) reads `process.env` by definition; that is
   where validated access is implemented (MC3 does not apply there).

## MC1 ‚Äî Render CLI output through the branded kit (`@moku-labs/common/cli`)

**Rule.** All human-facing CLI output goes through the branded console kit from
`@moku-labs/common/cli` (palette/`BRAND_PINK`, `box`, `spinnerFrameAt`, `makePalette`,
`createBrandConsole()`'s `info`/`warn`/`error`/`heading`/`check`/lockup, and the styled
`confirm`/`select` prompts). Do **not** hand-roll ANSI escape sequences, box-drawing characters, or
spinner animations, and do **not** use third-party prompt libraries for interactive CLI prompts.

**Rationale.** A single renderer keeps every moku CLI visually consistent (same palette, same box
style, same spinner, same prompt chrome) and centralizes terminal concerns (color capability
detection, width, no-TTY fallback). Hand-rolled ANSI drifts, breaks on dumb terminals, and
duplicates logic the kit already solves.

**Incorrect:**
```typescript
console.log("\x1b[35mDeploying‚Ä¶\x1b[0m");                       // raw ANSI color
console.log("‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê\n‚îÇ   Done    ‚îÇ\n‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò"); // hand-built box
const frames = ["‚Ýã", "‚Ýô", "‚Ýπ", "‚Ý∏"];                            // hand-rolled spinner frames
let i = 0; setInterval(() => process.stdout.write(`\r${frames[i++ % 4]}`), 80);
```

**Correct:**
```typescript
import { createBrandConsole, box, spinnerFrameAt } from "@moku-labs/common/cli";

const con = createBrandConsole();
con.heading("Deploy");
con.info("Deploying‚Ä¶");
process.stdout.write(`\r${spinnerFrameAt(frame++)} working`);
process.stdout.write(box("Done"));
```

**Detection guidance.**
- In scoped CLI source (`*/cli*`, `*/scripts/**`, CLI plugins), grep for raw ANSI escapes
  (`\x1b[` / `[` / `\033[` / `\e[`), box-drawing characters (`‚îå ‚îê ‚îî ‚îò ‚îÄ ‚îÇ ‚ï≠ ‚ïÆ ‚ï∞ ‚ïØ ‚ïê ‚ïë`), and
  spinner-frame arrays (Braille `‚Ýã‚Ýô‚Ýπ‚Ä¶` or the classic `| / - \` rotation).
- A file that prints chrome **without** importing from `@moku-labs/common/cli` is the signal. If the
  file already imports the kit and the raw sequence is inside a fallback the kit doesn't cover,
  downgrade to WARNING.
- Skip the brand-kit source itself (`*/common/src/cli/*`) ‚Äî see shared exceptions.

## MC2 ‚Äî Use `ctx.log` for logging, not raw `console.*`

**Rule.** Diagnostic and event logging in plugin/CLI/script source goes through `ctx.log`
(`ctx.log.info/warn/error/debug`), provided by `logPlugin` from `@moku-labs/common`. Do **not** use
raw `console.log` / `console.info` / `console.warn` / `console.error` / `console.debug` for
logging.

**Rationale.** `ctx.log` is structured (level + message + structured fields), routable (a single
sink decides format/destination), and testable (a mock `ctx` captures log calls). Scattered
`console.*` calls cannot be levelled, filtered, redirected, or asserted on, and they bypass the
family's logging configuration.

**Incorrect:**
```typescript
console.log("user logged in", userId);
console.error("send failed", err);
```

**Correct:**
```typescript
ctx.log.info("auth:login", { userId });
ctx.log.error("mailer:send-failed", { err });
```

**Detection guidance.**
- Grep scoped source for `console.<method>(` where method ‚àà `log|info|warn|error|debug|trace`.
- Allowed: test files; the single sink line marked `// @log-sink`; and human-facing CLI *output*
  that correctly goes through the brand kit (MC1) ‚Äî branded `con.info()`/`con.error()` is the
  renderer, not a logger, and is fine.
- A genuine ambiguity (is this user-facing output or a diagnostic?) ‚Üí WARNING, not BLOCKER.

## MC3 ‚Äî Use `ctx.env` for environment access, not raw `process.env`

**Rule.** Read environment variables through `ctx.env` (`ctx.env.get(name)` /
`ctx.env.require(name)`), provided by `envPlugin` from `@moku-labs/common`. Do **not** read
`process.env` directly in plugin/CLI/script source.

**Rationale.** `ctx.env` validates presence (`require` throws an actionable error when a variable is
missing instead of silently yielding `undefined`), centralizes parsing/defaults, and is mockable in
tests. Raw `process.env.FOO` is unvalidated, untyped, and couples domain code to the ambient
process.

**Incorrect:**
```typescript
const key = process.env.API_KEY;                 // undefined if unset ‚Äî no error
const port = Number(process.env.PORT ?? 3000);
```

**Correct:**
```typescript
const key = ctx.env.require("API_KEY");          // throws "[framework] Missing env API_KEY." if unset
const port = ctx.env.get("PORT") ?? 3000;        // validated accessor with a default
```

**Detection guidance.**
- Grep scoped source for `process.env`.
- Allowed: test files; and **env providers** ‚Äî the module backing `envPlugin` (`*/env/*`, a
  `*EnvProvider` export, or `env-provider.ts`) is *where* validated access is implemented and must
  read `process.env`.
- A legitimate **passthrough** ‚Äî e.g. spreading `process.env` into a spawned subprocess (a
  wrangler/CLI child) ‚Äî may mark that line with a `// @env-allow` comment; the hook skips marked
  lines (the same escape-hatch pattern as `// @log-sink` for MC2).
- Reading `process.env.NODE_ENV` in a build/tooling config is a gray area ‚Äî prefer `ctx.env` in app
  code, but a `*.config.ts` is already exempt (shared exception #1).

## When a project legitimately establishes a new exception

If a build needs another sanctioned `console.*` sink or another env-provider path, add it to the
"Shared exceptions" list above (and reference it from the `moku-common` SKILL's Anti-Patterns
section) so it is never re-flagged ‚Äî the same convention-baseline discipline used in
`house-style.md`.
